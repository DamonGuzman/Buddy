# COORD-STUDY: isolating raw model coordinate estimation

Controlled experiment (branch `exp/coord-study`, everything under
`eval/experiments/coord-study/`) that removes the entire app pipeline —
Electron capture, DIP mapping, overlay, turn machinery — and measures the
MODEL's raw `point_at` coordinate estimation against known-exact ground
truth, plus whether compositing visual references (gridlines / rulers /
fiducial markers) onto the screenshot improves it.

Motivation: live evals (docs/EVAL.md §7-§8) showed gpt-realtime-2.1 names the
right element ~100% of the time but its pixel coordinates drift
scene-dependently (median ~90-105px on 2048px images, worst 533px), while the
app's own coordinate mapping is proven exact (calibration 0px). Question:
is anything app-side off, and can prompt/image-side interventions fix it?

## 1. Setup

- **Harness**: `harness.mjs` — plain Node + `ws`, speaks the GA Realtime WS
  protocol directly (`wss://api.openai.com/v1/realtime?model=…`), same
  session/tool conventions as `src/main/realtime/session.ts` but with
  `output_modalities: ['text']` (no audio; ~10x cheaper, same tool-call path).
  One session per condition x layout; every target is its own user turn of
  [framing text, image, "point at the <target>"], mirroring production where
  the screenshot rides along on every ask. Tool calls are acked
  (`{"ok":true}`) without requesting a continuation.
- **Synthetic images** (exact ground truth): 2048x1152 JPEG q90, light
  background, generated by `render.ps1` (System.Drawing) from `layouts.json`.
  12 targets per layout: 4 corner buttons, 1 center, 2 edge, a dense cluster
  of 3 adjacent buttons (150px pitch), a 24px dot and a 16px square. Two
  layout variants (A, B) with different positions to avoid overfitting
  conclusions to one arrangement. Centers are exact by construction.
- **Real screenshot**: `capture-real.ps1` + `Capture.cs` (compiled helper;
  PowerShell's `CopyFromScreen` is AMSI-blocked on this machine) captures the
  physical 3840x2160 primary display and downscales to 2048x1152 — the app's
  exact capture geometry. 5 ground-truth targets hand-measured by 1:1 crop
  inspection with crosshair verification (`real-targets.json`, ±3px).
- **Scoring**: error = euclidean px distance from the target's true center;
  `hit` = prediction inside the element's actual bounds; thresholds ≤40px
  (the app's "near" radius at DIP scale) and ≤100px.
- **Stats honesty**: n = 12 targets/condition/layout (5 on real). Small — all
  numbers below are raw, no significance claims.

## 2. Conditions

| condition | image | framing / tool |
|---|---|---|
| baseline-plain | bare | minimal: dims + "pixel coords, origin top-left" |
| baseline-anchors | bare | production framing verbatim (corner anchors + fraction→pixel worked example + anti-center clause) |
| grid-100 | 100px semi-transparent gridlines, red axis labels every 200px | plain + "read position off the gridlines" |
| ruler-edge | tick marks all 4 edges (50px minor / 100px major), numbers every 200px | plain + "project onto the rulers" |
| fiducials | 9 red crosshairs at known positions, each labeled "(x,y)" | plain + "interpolate from the nearest markers" |
| normalized | bare | tool takes x,y as 0-1000 normalized (converted for scoring) |
| think-first | bare | model must first state the location in words (quadrant/fraction/landmark), then call point_at |

Model under test: `gpt-realtime-2.1` (production default). Transfer checks:
best conditions re-run on `gpt-realtime-2.1-mini`; sanity condition on a
strong non-realtime vision model via REST (`rest-sanity.mjs`, gpt-5.2,
strict-JSON coords) to see whether coordinate weakness is
realtime-family-specific.

## 4. Drift structure

Affine fits (gt→pred, per layout; identity = x:[1,0,0] y:[0,1,0]):

| condition/layout | x coefficients | y coefficients | raw RMS | residual RMS after affine |
|---|---|---|---:|---:|
| plain A | [1.0, 0.1, 17] | [0.0, 1.0, -34] | 146 | 124 |
| plain B | [0.9, 0.2, -69] | [0.0, 1.0, -63] | 149 | 101 |
| anchors A | [0.9, 0.1, 9] | [0.0, 1.0, -26] | 152 | 129 |
| anchors B | [0.9, 0.1, -20] | [0.1, 1.0, -145] | 149 | 111 |
| grid-100 A | [0.9, 0.1, 54] | [0.0, 0.9, 4] | 97 | 77 |
| grid-100 B | [1.0, 0.1, -28] | [0.0, 1.0, -18] | 103 | 83 |

Key structural findings (per-target error vectors):

- **Corners are nearly solved** (20-60px in every condition) — extreme
  landmarks anchor well. The center button is fine too.
- **The dense cluster displaces as a rigid group**: e.g. anchors/B predicted
  COPY/PASTE/CUT at exactly (-140,-20) off each — inter-element spacing
  preserved perfectly (150px pitch reproduced to the pixel) while the whole
  block sits a button-width away. The model reads relative layout correctly
  and mislocates the absolute frame — same signature as the live shop-page
  misses.
- **Small isolated icons drift worst** (100-280px): no nearby text/landmark
  to anchor on.
- **Coordinates are quantized**: predictions cluster on multiples of 10-50px
  ("round numbers"), an intrinsic output-precision floor.
- **Affine correction is confirmed unsound**: fitting a full per-scene affine
  only shrinks RMS ~15-30% (146→124, 149→101…), residuals stay ~80-130px.
  The drift is locally coherent but globally non-linear — matches the
  M8.6 decision to reject a display-profile correction.
- **The production anchor framing is a wash on neutral scenes** (81 vs 83
  median, within-40 38% vs 29%) — consistent with §8.1 where it helped
  toolbar chrome but hurt the shop page.
- **grid-100 doesn't move the median (84px) but halves the tail**: p90
  243→133, max 292→200, cluster median 237→128. The model uses gridlines to
  avoid gross mislocation but still can't do better than ~1 grid cell.

## 7. Conclusions (updated as results land)

1. **Nothing app-side is off.** On a bare WS harness with programmatically
   exact ground truth — no capture, no DPI/DIP mapping, no overlay — the raw
   model reproduces the live numbers (~80px median, ~300px tail vs live
   ~90-105px median, 533px worst). Combined with the 0px calibration, the
   entire pointing error is the model's image-space coordinate estimation.
2. **Anchor framing (shipped in M8.6) does not measurably improve neutral
   scenes**; it is not harmful either (38% vs 29% within-40 is noise at n=24).
3. **Gridline compositing is the most promising overlay so far**: it doesn't
   fix typical error but reliably clips the catastrophic tail (worst-case
   halved). Whether that's worth polluting the model's view of the screen
   depends on the remaining conditions.
4. Neither baseline nor any completed intervention approaches the bar that
   would change the product decision (median <40px or ≥70% within 40px).
   Element snapping (UIA/OCR label-matching, EVAL.md §8.4) remains the only
   credible route to strict-gate accuracy.

## 8. Reproduce / resume

```sh
cd eval/experiments/coord-study
npm install                         # ws only
powershell -EP Bypass -File render.ps1          # synthetic images
node harness.mjs --condition grid-100 --model gpt-realtime-2.1 --layouts A,B
node rest-sanity.mjs --model gpt-5.2            # REST sanity
node harness.mjs --condition baseline-anchors --model gpt-realtime-2.1 \
  --image images/real-plain.jpg --targets real-targets.json   # real screenshot
node analyze.mjs                                # summary table + affine fits
sh resume-when-quota.sh                         # poll quota, run remainder
```
